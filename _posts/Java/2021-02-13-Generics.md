### 1. 제네릭 도입 이유 (타입 안정성, Type Safety)

제네릭은 JDK 1.5부터 도입되었다. 그 전에는 제네릭을 사용하지 않았었다는 것입니다.

사용하지 않고도 잘 쓰고 있었는데, 왜 도입하게 되었을까요?

다음 코드는 아까 설명한 과일들을 담은 박스에 대한 코드입니다.

이 코드는 제네릭을 사용하지 않은 코드입니다.

```java
class FruitBox {
	List fruits = new AraayList();

	public void add(Object fruit) {
		fruits.addFruit(fruit);
	}

	public Object getFruit(int index) {
		return fruits.get(index);
	}
}
```

```java
public static void main(String[] args) {
	FruitBox applebox = new FruitBox();
	appleBox.addFruit(new Apple());
	
	Apple apple = (Apple) appleBox.getFruit(0);
	Pear pear = (Apple) appleBox.getFruit(0);	
}
```

JDK 1.5 이전에서는 여러 타입을 사용하는 대부분의 클래스나 메소드에서 인수나 반환값으로 Object 타입을 사용했습니다.

하지만 이 경우에는 반환된 Object 객체를 다시 원하는 타입으로 타입 변환해야 하며, 이때 오류가 발생할 가능성도 존재합니다. 예를 들어, 위에 코드와 같이 0번째 인덱스에는 배가 들어있는 줄 알았는데, 알고 보니 사과가 들어있었다. 라는 상황도 있을 수 있습니다.

이런 오류는 실행해보지 않는 이상 모를 것입니다. 실행하고 나서야 이런 버그가 있었다는 것을 알게 되는거죠. 문법적으로는 문제가 없지만 논리적인 자료형의 오류에 대한 검증이 컴파일 타임에 이루어지지 않음으로서 런타임에 오류가 발생하거나 또는 오류 사실을 알지 못하는 경우가 발생합니다.

```java
public static void main(String[] args) {
	FruitBox applebox = new FruitBox();
	appleBox.addFruit(new Apple());
	
	Apple apple = appleBox.getFruit(0);
	Pear pear = appleBox.getFruit(0);	// 컴파일 에러 발생

	applebox.addFruit(new Pear()); // 컴파일 에러 발생
}
```

애플 박스의 0번째를 가져오려고 하면 컴파일 에러가 발생합니다.

appleBox에는 Apple 클래스밖에 없으므로 Pear 클래스와 맞지 않기 때문이죠.

또, 하나의 장점이 있습니다.

애플 박스에 배 클래스를 넣으려고 하면 컴파일 에러가 발생합니다.

컴파일러가 배 클래스는 애플 박스가 받을 수 있는 클래스가 아니므로 경고를 띄어주는 겁니다.

이렇게 컴파일러는 강한 타입 체크를 해줍니다. 이것을 타입 안정성이라고 합니다.

컴파일러는 제네릭 클래스가 건네받은 타입 매개변수만을 사용하는 것을 보장해줍니다.

이것이 제네릭의 키 컨셉입니다. 컴파일러에게 사용하려는 타입을 알려 주면, 컴파일러가 모든 세부 정보를 처리합니다.

다음과 같이 제네릭을 사용하면 컴파일 시에 미리 타입이 정해지므로, 타입 검사나 타입 변환과 같은 번거로운 작업을 생략할 수 있게 됩니다.

이렇게 컴파일 시에 미리 타입 검사(type check)를 수행하면 다음과 같은 장점을 가집니다.

1. 클래스나 메소드 내부에서 사용되는 객체의 타입 안정성을 높일 수 있습니다.
2. 반환값에 대한 타입 변환 및 타입 검사에 들어가는 노력을 줄일 수 있고, 형변환이 없어지므로 가독성이 좋아집니다.

클래스에서 정의된 제네릭과 메소드에서 선언한 제네릭의 타입은 다른 타입이다.

- 컴파일 단계에서 오류가 검출된다. 즉, 컴파일팀에 논리적인 자료형의 오류에 대한 검증을 수행하여 런타임에 자료형에 안전한 코드를 실행합니다.
- 중복의 제거와 타입 안전성을 동시에 추구할 수 있게 되었다.

### 2. 제네릭이란?

제네릭 클래스는 다음 형식으로 정의됩니다.

```java
class name<T1, T2, ..., Tn> { /* ... */ }
```

다음 코드는 전에 설명한 FruitBox 클래스입니다.

```java
class FruitBox {
	List fruits = new AraayList();

	public void add(Object fruit) {
		fruits.addFruit(fruit);
	}

	public Object getFruit(int index) {
		return fruits.get(index);
	}
}
```

위 코드는 다음과 같이 제네릭으로 작성할 수 있습니다.

```java
class FruitBox<T> {
	List<T> fruits = new AraayList<>();

	public void add(T fruit) {
		fruits.addFruit(fruit);
	}

	public T getFruit(int index) {
		return fruits.get(index);
	}
}
```

제네릭 클래스의 인스턴스 생성은 다음과 같습니다.

```java
FruitBox<Apple> appleBox = new FruitBox<>();
```

제네릭 타입의 호출은 기존 메소드 호출과 비슷합니다. 메소드에 인자를 넘기는 것 대신에, 클래스에 타입 인자를 넘깁니다.

제네릭이란, 클래스나 메소드에서 사용할 내부 데이터 타입을 외부에서 지정하는 기법입니다.

즉, 클래스를 정의 할 때는 데이터 타입을 확정하지 않고 인스턴스를 생성할 때 데이터 타입을 지정하는 기법이다.

클래스 선언에 타입 매개 변수가 쓰이면, 이 클래스를 제네릭 클래스라 합니다.

다음은 제네릭 클래스의 형태입니다.

다음과 같은 박스 클래스가 있습니다.

이 박스 클래스는 리스트 형태의

```java
class FruitBox<T> {
		T fruit;
}

FruitBox<Apple> box = new FruitBox<>();
```

위처럼 제네릭 클래스를 생성할 때 사용할 실제 타입을 명시하면, 내부적으로는 정의된 타입 변수가 명시된 실제 타입으로 변환되어 처리됩니다.

사용하는 필드는 지정한 타입이 된다고 생각하시면 편합니다.

하지만 실제로 지정한 타입이 된다는 것은 아닙니다. 이것은 뒤의 Type Erasure에서 설명하겠습니다.

### 3. 제네릭 메소드

클래스 전부가 아닌 특정 메소드에 대해서만 제네릭으로 선언하는 것을 허용합니다.

제네릭 메소드는 타입 매개변수를 반환 타입 전에 지정해주면 됩니다.

```java
public <T> void test(T t) {
	...
}
```

여기서 <T> 는  앞으로 test메소드에서 T라는 매개변수를 사용하겠다는 것을  선언하는 것입니다. 이것으로 test() 메소드는 지역변수를 선언한 것처럼 T 타입의 변수를 사용할 수 있습니다. 이 타입 매개변수의 범위는 메서드의 블록 이내로 한정됩니다.

제네릭 메소드는 제네릭 클래스뿐만 아니라, 제네릭 클래스가 아니어도 정의가 가능합니다. 즉, 제네릭 메소드는 클래스에 독립적으로 정의할 수 있습니다.

```java
public class GenericMethod {
	public <T> void test(T t) {
		System.out.println(t.getClass().getName());
	}

	public static void main(String[] args) {
		GenericMethod generic = new GenericMethod();
		generic.test("");
		generic.test(123);
	}
}

// java.lang.String
// java.lang.Integer
```

위 예제에서 클래스는 제네릭이 아니고, 오직 메소드만이 타입 매개변수를 가지고 있습니다.

### 제네릭 클래스와 제네릭 메소드 간의 타입 매개변수 차이점

제네릭 클래스는 객체를 생성할 때 반드시 타입 파라미터의 값을 지정해야 합니다.

제네릭 메소드는 타입 파라미터를 지정할 필요가 없습니다. 왜냐하면 타입 매개변수 참조로 매개변수 타입을 알려줄 것이기 때문입니다.

따라서 일반 메서드처럼 test()를 호출할 수 있습니다. test()가 무한히 오버로드 된 것 같습니다.

예를 들어, 위의 메소드 `generic.test(123)` 은 따로 타입을 지정해주지 않았습니다. 하지만, 잘 동작합니다.

왜냐하면, 123은 오토박싱되어 프리미티브 타입을 해당 래퍼 객체에 래핑하기 때문입니다.

### 4. 타입 매개변수의 제한

다음은 배열에서 가장 작은 객체를 찾는 코드입니다.

![https://s3-us-west-2.amazonaws.com/secure.notion-static.com/8daece1d-72a5-426f-966d-24073c0104fd/Untitled.png](https://s3-us-west-2.amazonaws.com/secure.notion-static.com/8daece1d-72a5-426f-966d-24073c0104fd/Untitled.png)

만약 배열이 null이거나 비어있다면 null을 반환합니다. 첫 번째 배열의 값을 0으로 잡고, 배열의 루프를 돕니다. 만약 `small` 의 값이 배열의 현재 인덱스의 값보다 크다면 `small` 을 배열의 현재 인덱스값으로 할당합니다. 그 후 `small` 을 리턴합니다.

위 코드가 정상적으로 작동할까요?

절대로 아닙니다.

왜냐하면 주어진 타입 매개변수 `T`가  반드시 `compareTo` 메소드를 갖고 있다고 할 수 없기 때문입니다.

그러면 T가 반드시 compareTo 메소드를 갖고 있게 하려면 어떻게 해야할까요? 바로 T가 Comparable을 상속받게 하면 됩니다.

그럴 때 사용하는 것이 타입매개변수의 제한입니다.

타입 매개변수의 제한에는 2가지 종류가 있습니다.

- T extends Comparable : T는 반드시 Comparable을 상속받은 객체여야 합니다.
- T super Comparable : T는 반드시 Comparable의 상위 클래스여야 합니다.

즉, 다음과 같이 T가 Comparable을 상속받도록 한다면 T는 반드시 compareTo 메소드를 갖게 됩니다.

![https://s3-us-west-2.amazonaws.com/secure.notion-static.com/e45be819-a873-4ca9-9161-6349df943f12/Untitled.png](https://s3-us-west-2.amazonaws.com/secure.notion-static.com/e45be819-a873-4ca9-9161-6349df943f12/Untitled.png)

다음과 같이 여러 타입 매개변수를 지정할 수도 있습니다.

![https://s3-us-west-2.amazonaws.com/secure.notion-static.com/b5786b6b-6ce4-4ed3-9f02-319fa34b1759/Untitled.png](https://s3-us-west-2.amazonaws.com/secure.notion-static.com/b5786b6b-6ce4-4ed3-9f02-319fa34b1759/Untitled.png)

- 공변(covariant)
- PECS

```java
public <T> void print(T t) { // 메인 메소드에서 <T> 로 전달, <T>에서 print 인수 T로 전달
	sout(t)
}

public <T> void print(List<T extends Fruit> list) { } // x
public <T extends Fruit> void print(List<T> list) { } // o

// 와일드카드를 사용하면 앞에 타입 매개변수를 작성하지 않아도 된다
public void print(List<? extends Fruit> list) { } // x

// 와일드 카드는 타입 매개변수로 올 수 없다
public <?> void print(List<?> t) { } // 컴파일 에러

// 물론 다음의 코드도 불가하다
public <? extends Fruit> void print(List<?> t) { } // 컴파일 에러
```

알다시피 A 객체가 B 객체의 상위 클래스라면, B 객체를 A 객체라고 할 수 있습니다. 예를 들어, Integer 객체를 Object 객체라고 할 수 있습니다.

```java
Object someObject = new Object();
Integer someInteger = new Integer(10);
someObject = someInteger;   // OK
```

그러므로 다음 메소드에서 Number는 Integer, Double의 상위 클래스이므로 사용가능합니다.

```java
public void someMethod(Number n) { /* ... */ }

someMethod(new Integer(10));   // OK
someMethod(new Double(10.1));   // OK
```

이것은 제네릭에서도 똑같습니다. Number를 타입 파라미터로 갖고 있는 제네릭 타입은 만약 Number가 인자의 상위 클래스라면 호출 가능합니다.

```java
Box<Number> box = new Box<Number>();
box.add(new Integer(10));   // OK
box.add(new Double(10.1));  // OK
```

하지만 다음과 같은 코드는 불가능합니다.

```java
public void boxTest(Box<Number> n) { /* ... */ }

Box<Integer> box = new Box<>();
boxTest(box);
```

왜냐하면, Box<Integer>는 Box<Number>의 하위 클래스가 아니기 때문입니다.

![https://s3-us-west-2.amazonaws.com/secure.notion-static.com/5f90d0f6-5f94-4d30-b132-939837cf577c/Untitled.png](https://s3-us-west-2.amazonaws.com/secure.notion-static.com/5f90d0f6-5f94-4d30-b132-939837cf577c/Untitled.png)

클래스나 인터페이스를 상속하여 서브타입을 생성할 수 있습니다. 하위 클래스에 선언된 타입 파라미터들과 상위 클래스에 선언된 타입 파라미터들의 관계는 extends 와 implements 절에서 결정됩니다.

ArrayList<E> 는 List<E>를 구현합니다. List<E>는 Collection<E>를 상속받습니다. 그러므로 ArrayList<String>은 List<String>, Collection<String>의 서브타입입니다.

### 4. Type Erasure

제네릭을 구현하면, 자바 컴파일러는 타입 이레이져를 적용합니다.

타입 이레이져는 다음과 같은 일을 합니다.

- 제네릭 타입의 모든 타입 파라미터를 경계 타입, 만약 경계가 없다면 Object로 치환합니다. 그러므로, 생성된 바이트코드는 오직 기존의 클래스, 인터페이스, 메소드만을 가집니다.
- 만약 타입 안정성을 유지해야 한다면 캐스팅을 삽입합니다.

타입 이레이져는 파라미터화된 타입에 대한 새로운 클래스를 생성하지 않는 것을 보장해주빈다. 결과적으로 제네릭은 런타임 오버 헤드를 발생시키지 않습니다.

타입 이레이져는 컴파일 타임에만 타입 제약을 적용하고, 런타임에는 타입 정보를 버리는 절차를 의미합니다.

매개변수화 타입, 즉 타입 매개변수가 가진 클래스 정보는 컴파일러에 의해 없어집니다. 이것을 `Type Erasure` 라고 부릅니다.

다음과 같은 제네릭 Box 클래스가 있습니다.

```java
public class Box<T> {
    private List<T> objects = new ArrayList<>();
    
    public void add(T object) {
        objects.add(object);
    }
    
    public T get(int index) {
        return objects.get(index);
    }
}
```

컴파일러는 다음과 같이 정규 타입 매개변수를 Object로 변경합니다.

```java
public class Box {

   private List objects = new ArrayList();

   public void add(Object object) {
      this.objects.add(object);
   }

   public Object get(int index) {
      return this.objects.get(index);
   }
}
```

그러므로 컴파일러는 타입 안정성을 보장하고, 런타임 에러를 방지합니다.

Object 타입으로 가지고 있다가 사용할 때 Casting 하여 원하는 클래스로 반환한다.

```java
public class generic/Box {

	private Ljava/util/List; objects

	//	public 메소드명(파라미터 타입) 반환 타입

  public add(Ljava/lang/Object;)V // 파라미터의 타입이 T가 아닌 Object
		...
    INVOKEINTERFACE java/util/List.add (Ljava/lang/Object;)Z (itf)
		...

  public get(I)Ljava/lang/Object; // 반환 타입이 T가 아닌 Object
		...
    INVOKEINTERFACE java/util/List.get (I)Ljava/lang/Object; (itf)
		...
}
```

런타임에는 형식 타입(Formal Type)은 Bound 되었을 경우는 바운드 타입으로, Bound 되지 않았을 경우에는 Object로 바뀝니다.

예를 들어, 아래와 같은 형변환  T를 가진 제네릭 클래스가 있습니다.

Container는 Integer나 BigDecimal 등의 Number 클래스를 갖는 클래스입니다.

```java
class Container<T extends Number> {
    private T value;    
    public Container(T value) {
        this.value = value;
    }   
    public T get() {
        return value;
    }
}
```

컴파일러는 위의 클래스에서 타입 정보를 삭제하여, 아래의 클래스와 같이 만듭니다.

```java
class Container {
    private Number value;
    public Container(Number value) {
        this.value = value;
    }
    public Number get() {
        return value;
    }
}
```

타입 정보를 실제로 삭제하는 것은 아니고, 타입 매개변수는 그 상한경계의 타입(위에선 Number)로 변경됩니다. 이렇게 타입 정보는 컴파일러에 의해 소거되기 때문에, 실행시에는 타입 정보를 취득할 수 없습니다.

타입 이레이져는 제네릭을 사용하는 Java 애플리케이션이 제네릭이 도입되기 전에 생성된 Java 라이브러리 및 애플리케이션과 바이너리 호환성을 유지하도록 해줍니다.

예를 들어, Box<String>은 Box라는 raw type으로 변형됩니다. raw type은 타입 매개변수를 제외한  제네릭 클래스 혹은 인터페이스의 이름입니다. 이건 런타임에 사용되고 있는 제네릭 클래스의 실제 타입 매개변수를 찾을 수 없다는 것을 의미합니다.

그러므로 다음 연산은 불가능합니다.

```java
public class MyClass<E> {
    public static void myMethod(Object item) {
        if (item instanceof E) {  //Compiler error
            ...
        }
        E item2 = new E();       //Compiler error
        E[] iArray = new E[10];  //Compiler error
        E obj = (E)new Object(); //Unchecked cast warning
    }
}
```

위 코드는 런타임에 의미가 없습니다. 왜냐하면 컴파일러가 컴파일 타임에 실제 타입 매개변수에 대한 모든 정보를 지우기 때문입니다.

제네릭을 사용하지 않던 코드와 제네릭 코드와의 호환성을 위해 타입 이레이져가 존재합니다. raw type을 사용는 것은 어떤 이유에서든 나쁜 프로그래밍 방법으로 고려되니, 가능하면 피해야만 합니다.

이에 따라서 하나 더 알아두시면 좋을게 있는데요.

바로 타입 매개변수의 클래스를 생성할 수 없다는 것입니다.

즉, `new T()` 할 수 없습니다.

타입 파라미터는 소거되기 때문에, T의 인스턴스를 생성할 수 없습니다.

마찬가지로 `new T[]` 도 할 수 없습니다.

제네릭 클래스의 경우, 다음과 같이 매개변수 T를 타입으로 하는 인스턴스를 생성할 수 있습니다.

```java
class Sample<T> {
    public List<T> createList(int size) {
        //TのArrayListを生成せいせいできる
        return new ArrayList<T>(size);
    }
}
```

하지만, 타입 매개변수를 요소로 하는 배열을 생성할 수 없습니다.

```java
class Sample<T> {   
    public T[] createArray(int size) {
        return new T[size];
    }   
}
```

위에서 설명한대로, 타입 매개변수 T의 클래스 정보는 실행시에 소거되기 때문입니다. 제네릭 클래스의 경우, 클래스 정보를 포함하지 않는 원형을 생성하기 때문에, 실행시에 클래스 정보를 필요로 하지 않습니다. 한편, 배열의 생성에는 실행시에 클래스 정보가 필요하기 때문에(종학히는 배열을 생성하는 바이트코드가 클래스를 필요로 합니다), 타입 맥변수 T를 원소로 하는 배열은 생성할 수 없습니다.

```java
public class Generic {
    public static void main(String[] args) {
        int[] ints = new int[5];
        String[] strings = new String[10];
    }
}
```

### 제네릭의 제거 시기

자바 코드에서 선언되고 사용된 제네릭 타입은 컴파일 시 컴파일러에 의해 자동으로 검사되어 타입 변환됩니다.

그리고서 코드 내의 모든 제네릭 타입은 제거되어, 컴파일된 class 파일에는 어떠한 제네릭 타입도 포함되지 않게 됩니다.

이런 식으로 동작하는 이유는 제네릭을 사용하지 않는 코드와의 호환성을 유지하기 위해서입니다.

### 제네릭 클래스는 staic문(static 변수, static 메소드, static 초기화 블럭)에 사용할 수 없다.

정규 타입 매개변수의 실제 타입 매개변수는 외부에서 제네릭 클래스 타입의 레퍼런스를 선언하거나 인스턴스를 생성할 때 결정이 되는데, static문들든 인스턴스의 생성과는 무관하게 존재하므로 실제 타입을 알 수 없다.

### 제네릭에서 Primitive Type을 사용할 수 없는 이유

제네릭에서 다음과 같이 타앱 매개 변수로 Primitive Type을 사용할 수 없다.

```java
List<int> list = new ArrayList<>(); // 컴파일 에러
```

어째서 primitive type이 작동하지 않는지 알기 위해선, 먼저 제네릭은 컴파일 타임 기능이라는 것을 기억해야 한다. 타입 매개변수는 지워지고, 모든 제네릭 타입은 Object 타입이 됩니다.

```java
// 제네릭 클래스의 메소드
boolean add(E e)

// 컴파일 후
boolean add(Object e)
```

그러므로 타입 매개변수는 반드시 Object로 바뀔 수 있어야 합니다. 하지만 primitive type은 Object를 상속받지 않았기 때문에 primitive 타입을 타입 매개변수로 사용할 수 없습니다.

그러나, 자바는 오토박싱과 언박싱과 함께 primitive type에 대한 박싱된 타입을 제공합니다.  그러므로 만약 Integer 클래스를 담은 리스틀 만들고 싶다면, 이 wrapper 클래스를 이용합니다.

```java
List<Integer> list = new ArrayList<>();
list.add(17);
int first = list.get(0);
```

컴파일된 코드는 다음과 같습니다.

```java
List list = new ArrayList<>();
list.add(Integer.valueOf(17));
int first = ((Integer) list.get(0)).intValue();
```

### 6. Wildcar Type

### Unbounded Wildcards

Unbounded Wildcard가 유용한 2가지 경우가 있다.

- Object 클래스의 메소드를 사용하여 메소드를 구현하는 경우
- 타입 파라미터에 의존하지 않는 제네릭 클래스의 메소드를 사용할 때. 예를 들어, List.size 혹은 List.claer.

### 여러 리스트 간의 관계

![https://s3-us-west-2.amazonaws.com/secure.notion-static.com/fa591576-6a9c-46ce-98a2-612d3f197122/Untitled.png](https://s3-us-west-2.amazonaws.com/secure.notion-static.com/fa591576-6a9c-46ce-98a2-612d3f197122/Untitled.png)

### Wildcard Guidelines

- `in` variables는 상한 와일드카드로 정의되어야 한다.
- `out` variables는 하한 와일드카드로 정의되어야 한다.
- Object 클래스에 정의된 메소드를 사용하여 `in` 변수에 접근할 수 있는 경우 Unbounded 와일드카드를 사용한다.
- `in` , `out` 변수 둘 다 접근해야 한다면 와일드 카드를 사용하면 안된다.

이 가이드라인은 메소드 리턴 타입에는 적용하지 않습니다. 리턴 타입에 와일드 카드를 사용하는 것은 피해야 합니다.  그것은 프로그래머가 코드를 사용하여 와일드카드를 처리하도록 합니다.

List<? extends ...?> 은 read-only로 생각할 수 있습니다. 엄격한 의미에서는 read-only가 아닙니다. 하지만 새 요소를 저장하거나 기존 요소를 변경할 수 없기 때문에 그렇게 생각할 수 있습니다.

다음 메소드를 보자.

```java
public static void printList(List<Object> list) {
    for (Object elem : list)
        System.out.println(elem + " ");
    System.out.println();
}
```

printList()는 어떤 타입의 리스트이든 print한다. 하지만 오직 Object의 리스트 인스턴스만을 프린트한다. List<Integer>, List<Stirng> 등은 프린트 할 수 없다. 왜냐하면, 그것들은 List<Object>의 하위타입이 아니기 때문이다. printList 메소드를 사용하려면 List<?>를 사용해야 한다. 왜냐하면, 특정 타입 A의 리스트 List<A>는 List<?>의 하위 타입이기 때문이다.

```java
public void wildcardHasAnyType() {
    List<String> stringList 
			= new ArrayList<>();

    List<Integer> integerList 
			= new ArrayList<>();

    boolean b1 = contains(stringList, "a");
    boolean b2 = contains(integerList, 3);
}

boolean contains(List<?> list, Object o) {
    return list.contains(o);
}
```

List<?>는 어떤 클래스의 원소를 가진 리스트를 나타냅니다. List<?>에는 임의의 List를 대입할 수 있습ㄴ디ㅏ.

이것은 공변의 성질에 의해 List<Object>에는 List<String>을 도입할 수 없었던 것과 대조적입니다. 이렇게 와일드카드를 이용하면 제네릭의 불변 제약을 느슨하게 할 수 있습니다.

와일드 카드는 클래스 선언에는 사용할 수 없다.

### List<?>에 원소를 추가할 수 없다

List<?>는 임의의 원소를 가진 List이지만, 구체적인 원소 타입에 대해서는 부정확합니다. 그래서 다음 코드와 같이 여러 타입의 값이 들어올 수도 있습니다. 그래서 List에 어떤 타입이 추가될지 모릅니다. 따라서 null값을 제외한 어떤 값도 전달할 수 없습니다. 따라서 아래와 같은 코드는 컴파일에러가 발생합니다.

```java
public void foo(List<?> anyList) {
    anyList.add("error"); //コンパイルエラー
		anyList.add(1);
}
```

보다 일반적으로 말하자면, 타입 매개변수가 인수에 나타나는 비경계 와일드카드 메소드를 호출할 수 없습니다.

```java
interface UnaryFunction<T> {
    public T apply(T value);
}

class Illegal {
    public void foo(UnaryFunction<?> f) {
        f.apply("a"); //コンパイルエラー
    }
}
```

### List<?>에서 get한 원소는 Object이다

비경계 와일드카드의 원소는 어떤 타입도 될 수 있기 때문에, 꺼낸 원소 타입은 가장 범용적인 형인 Object 입니다.

```java
public void bar(List<?> list) {
    Object o = list.get(0);
    String s = list.get(0); //コンパイルエラー
}
```

### List<? extends T>는 공변성을 가진다

List<? extends T>는 상한적 와일드카드 타입입니다. 이것은 타입 T의 어떤 서브클래스를 원소로 하는 List를 나타냅니다. 상한적 와일드 클래스 타입을 사용하면 다음 코드와 같이, 불변이었던 제네릭에 공변성을 도입할 수 있습니다.

```java
public void foo() {
    List<Number> numList = new ArrayList<>();
    List<? extends Number> wildCardNumList = new ArrayList<>();
    List<Integer> intList = new ArrayList<>();

    wildCardNumList = intList; //OK
    numList = intList; //コンパイルエラー
}
```

### List<? super T>는 반변성을 가진다

List<? super T>는 하한적 와일드 카드입니다. 이것은 타입 T의 어떤 슈퍼 클래스를 원소로 하는 List라는 의미입니다. 하한적 와일드 카드를 사용하면 아래 코드처럼, 불변이었던 제네릭에 반변성을 도입할 수가 있습니다.

```java
public void bar() {
    List<Integer> intList = new ArrayList<>();
    List<? super Integer> wildCardIntList = new ArrayList<>();
    List<Number> numList = new ArrayList<>();

    wildCardIntList = numList; //OK
    intList = numList; //コンパイルエラー
}
```

지정된 제네릭 타입은 유연하지 않고 재사용하기 어렵습니다. 그래서 나온 것이 `wildcard` 입니다.

제한되지 않은 와일드카드는 메소드에서 formal 매개변수를 선언할 때 더 중요합니다.

![https://s3-us-west-2.amazonaws.com/secure.notion-static.com/57d2744c-cce6-48c8-82c5-2d995b770bf2/Untitled.png](https://s3-us-west-2.amazonaws.com/secure.notion-static.com/57d2744c-cce6-48c8-82c5-2d995b770bf2/Untitled.png)

위 코드에서 runOne 메소드는 오직 지정된 타입 `Animal` 클래스만을 매개변수로 받는 것을 알 수 있습니다.  그리고 runTwo 메소드는 모든 타입을 매개변수로 받을 수 있습니다. 그러므로 더 유연합니다.

extends 키워를 사용한 곳에서 Animal 클래스의 하위 클래스들을 파라미터로 넘겨줄 수 있습니다.

여러 upper bound 들을 콤마로 구분해서 지정할 수도 있습니다.

### super

파라미터화된 타입(제네릭 클래스)는 반드시 지정한 타입 혹은 그 상위클래스여야 합니다.

![https://s3-us-west-2.amazonaws.com/secure.notion-static.com/d24b4c98-dabd-451c-9c1b-3336efcab7da/Untitled.png](https://s3-us-west-2.amazonaws.com/secure.notion-static.com/d24b4c98-dabd-451c-9c1b-3336efcab7da/Untitled.png)

다음과 같은 클래스 계층도가 있습니다.

![https://s3-us-west-2.amazonaws.com/secure.notion-static.com/00cab4dd-ca79-4b6d-866e-9257b8237f0d/Untitled.png](https://s3-us-west-2.amazonaws.com/secure.notion-static.com/00cab4dd-ca79-4b6d-866e-9257b8237f0d/Untitled.png)

상한적 와일드 카드

extends 뒤에 클래스를 붙이면 해당 와일드 카드는 클래스 혹은 그 하위 클래스 전부를 의미합니다.

![https://s3-us-west-2.amazonaws.com/secure.notion-static.com/28fdb381-9027-4737-a391-5b1c8d041562/Untitled.png](https://s3-us-west-2.amazonaws.com/secure.notion-static.com/28fdb381-9027-4737-a391-5b1c8d041562/Untitled.png)

List<? extends Fruit> 클래스를 사용하여, Fruit 클래스 혹은 그 하위 클래스를 지정한 List형으로 선언 가능합니다.

```java
List<? extends Fruit> basket = new ArrayList<Fruit>(); // 1
List<? extends Fruit> basket = new ArrayList<Melon>(); // 2
List<? extends Fruit> basket = new ArrayList<Lemon>(); // 3
```

다음으로 baseket에서 원소를 꺼낼 떼, 그 서브 클래스를 지정할 수 있는가를 생각해보겟습니다.

List<? extends Fruit> 은 적어도 아래 3개의 List로 변환 가능합니다.

```java
1. ArrayList<Fruit>
2. ArrayList<Melon>
3. ArrayList<Lemon>
```

따라서 원소를 꺼낼 때, 몇 가지 계열에 파생되어 있는 서브 클래스를 지정할 수가 없습니다.

```java
List<? extends Fruit> basket = ...

// NG 컴파일 에러, 멜론형으로 변환 불가
Melon fruit = basket.get(i);
```

만약 `? extends Fruit` 을 서브 클래스로 변환하려고 했다면, 아래와 같은 모순이 발생한다.

```java
List<Melon> melonList = new ArrayList<Melon>();
melonList.add(new Melon());

List<? extends Fruit> basket = melonList;

// 서브 클래스로의 변환이 가능하다면 Lemon으로 가져올 수 있다
Lemon fruit = basket.get(i);
```

따라서 서브 클래스로의 변환이 허용되지 않습니다.

따라서 `? extends T` 에서 T의 서브 클래스로의 변환은 불가능합니다.

다음으로 `List<? extends Fruit> fruits` 로 원소를 추가하는 것을 생각해봅시다.

`? extends Fruit` 에서 `Fruit`으로 변환은 가능하지만, 다음과 같이 `fruits` 에 `Fruit` 오브젝트를 추가할 수 있는 것처럼 보입니다.

하지만 다음을 허용한다면 아래와 같이 모순이 발생합니다.

```java
List<? extends Fruit> fruits = new ArrayList<Melon>();
Fruit fruit = new Lemon();
fruits.add(fruit);
```

따라서, 구체적인 T 혹은 T의 서브 클래스로부터 `? extends T` 로의 변환은 불가능합ㄴ디ㅏ.

이것은 List 형을 상한적 와일드카드로 선언하면, 원소를 추가하는 것이 불가능하다는 것을 의미합니다.

### 하한적 와일드 카드

```java
Generic<? super T>
```

타입 매개변수를 위와 같이 지정하면 그 파라미터는 하한적 와일드 카드형입니다.

`? super T` 형은 T 혹은 T의 상위 클래스 전부를 의미합니다.

![https://s3-us-west-2.amazonaws.com/secure.notion-static.com/6a07505d-a91e-43c2-bd70-29f5b00ac795/Untitled.png](https://s3-us-west-2.amazonaws.com/secure.notion-static.com/6a07505d-a91e-43c2-bd70-29f5b00ac795/Untitled.png)

`List<? super Melon>` 은 Fruit 클래스 혹은 그 상위 클래스로 타입 매개변수를 선언할 수 있습니다.

```java
List<? super Melon> baskets = new ArrayList<Melon>(); // 1
List<? super Melon> baskets = new ArrayList<Fruit>(); // 2
List<? super Melon> baskets = new ArrayList<Food>(); // 3
List<? super Melon> baskets = new ArrayList<Object>(); // 4
```

## 7. Restrictions on Generics

### 타입이 타입 파라미터인 스태틱 필드를 선언할 수 없다

클래스의 스태틱 필드는 클래스의 모든 non-static 객체들이 공유하는 클래스 레벨 변수입니다. 따라서, 타입 파라미터의 스태틱 필드는 허용되지 않습니다. 예를 들어, 다음과 같은 클래스가 있습니다.

```java
public class MobileDevice<T> {
    private static T os;

    // ...
}
```

만약 타입 파라미터의 스태틱 필드가 허용된다면, 아래의 코드는 혼란스럽게 됩니다.

```
MobileDevice<Smartphone> phone = new MobileDevice<>();
MobileDevice<Pager> pager = new MobileDevice<>();
MobileDevice<TabletPC> pc = new MobileDevice<>();
```

왜냐하면 스태틱 필드 `os` 는 `phone`, `pager`, `pc` 에서 공유되기 때문입니다. 실제 os의 타입은 무엇일까요? `os` 는 동시에 `Smartphone`, `Pager`, `TabletPC` 가 될 수 없습니다. 그러므로 타입 파라미터의 스태틱 필드를 선언할 수 없습니다.

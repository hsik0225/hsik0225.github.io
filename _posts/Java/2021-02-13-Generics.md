### 1. 제네릭 클래스

먼저 제네릭의 정의에 대해서 소개하겠습니다.

제네릭이란, 클래스나 메소드에서 사용할 내부 데이터 타입을 외부에서 지정하는 기법입니다.

즉, 클래스를 정의할 때는 데이터 타입을 확정하지 않고 인스턴스를 생성할 때 데이터 타입을 지정하는 기법이다.

글로 보려니 조금 어려운 감이 있습니다.

간단하게 제네릭 클래스를 정의하는 방법을 얘기하며, 뒤에서 코드로 설명하겠습니다.

클래스 선언에 타입 매개 변수가 쓰이면, 이 클래스를 제네릭 클래스라 합니다.

다음은 제네릭 클래스를 정의하는 방법입니다.

여기서 클래스 뒤에 괄호로 들어있는 변수들이 타입 매개변수입니다.

```java
class name<T, U, ...> { /* ... */ }
```

name에는 클래스 이름을 넣습니다. 그 뒤로, 원하는 수만큼 타입 매개변수를 선언할 수 있습니다.

다음은 제네릭 클래스의 예시입니다. FruitBox는 T 타입의 과일들을 가진 리스트를 가지고 있으며, add로 과일을 추가할 수 있고, get으로 해당 인덱스의 과일을 가져올 수 있습니다.

```java
class FruitBox<T> {
    private T fruit;
    
    public FruitBox(T fruit) {
        this.fruit = fruit;
    }
    
    public T getFruit() {
        return fruit;
    }
}
```

위 FruitBox 클래스의 인스턴스 생성은 다음과 같습니다.

```java
FruitBox<Apple> appleBox = new FruitBox<>(); // generic type

FruitBox appleBox = new FruitBox(); // raw type
```

- FruitBox 클래스의 T, U, .. 를 타입 매개변수(type parameter)라고 합니다.
- 클래스에 타입이 붙은 FruitBox<Apple>을 매개변수화 타입이라고 합니다.
- 제네릭 타입이 타입을 제외하고 선언하면 raw type이라고 합니다.
- FruitBox 클래스에 전해진 타입 String을 실제 타입 매개변수(actual type parameter)라고 합니다.

제네릭 타입의 호출은 기존 메소드 호출과 비슷합니다. 메소드에 인자를 넘기는 것 대신에, 클래스에 타입 인자를 넘깁니다. 위 코드를 다시 볼까요?

```java
FruitBox<Apple> appleBox = new FruitBox<>();
```

appleBox의 인스턴스를 생성할 때 전해진 실제 타입 매개변수가 FruitBox에 전해집니다. 메소드는 메소드 이름(매개변수)로 보낸다면, 제네릭은 메소드 이름<타입 매개변수>로 인자를 전달합니다.

위처럼 제네릭 클래스를 생성할 때 실제 타입을 명시하면, 내부적으로는 정의된 타입 변수가 명시된 실제 타입으로 변환되어 처리됩니다.

Apple 타입을 전달받은 FruitBox 클래스는 다음과 같이 변경된다고 생각할 수 있습니다.

```java

class FruitBox<Apple> {
    private Apple fruit;
    
    public FruitBox(Apple fruit) {
        this.fruit = fruit;
    }
    
    public Apple getFruit() {
        return fruit;
    }
}
```

클래스 내부에서 사용하는 타입 매개변수는 지정한 타입이 된다고 생각하시면 편합니다. 하지만 실제로 지정한 타입으로 변경되는 것은 아닙니다. 이것은 그렇게 생각하면 편하다는 것입니다. 이것은 뒤의 Type Erasure에서 설명하겠습니다.

### 2. 제네릭 도입 이유

간단히 제네릭 클래스에 대해서 알아보았으니, 이제 제네릭을 왜 사용하는지 알아보겠습니다.

다음과 같이 사과, 바나나 클래스와 과일 하나를 상자에 담을 수 있는 Box 클래스가 있습니다.

```java
class Apple {}

class Banana {}

class FruitBox {
	private Object fruit;

	public FruitBox(Object fruit) {
		this.fruit = fruit;
	}

	public Object getFruit() {
		return fruit;
	}
}
```

다음은 FruitBox 클래스를 사용하는 코드입니다.

appleBox와 BananaBox를 만들어서 각각의 과일을 담고, 가져올 수 있습니다. 성공적으로 작동합니다.

```java
public static void main(String[] args) {
	FruitBox appleBox = new FruitBox(new Apple());
	FruitBox bananaBox = new FruitBox(new Banana());
	
	Apple apple = (Apple) appleBox.getFruit();
	Banana banana = (Banana) bananaBox.getFruit();	
}
```

이번에도 각각의 과일을 담고, 가져오는 코드입니다.

```java
public static void main(String[] args) {
    FruitBox appleBox = new FruitBox(new Apple());
    FruitBox bananaBox = new FruitBox(new Banana());

    Apple apple = (Apple) appleBox.getFruit();
    Banana banana = (Banana) appleBox.getFruit();
}
```

이번에는 작성자가 실수를 했습니다. appleBox에서 가져온 과일을 바나나로 캐스팅을 해버린 것입니다. 이런 오류는 실행해보지 않는 이상 모를 것입니다. 실행하고 나서야 이런 버그가 있었다는 것을 알게 되는거죠. 문법적으로는 문제가 없지만 논리적인 자료형의 오류에 대한 검증이 컴파일 타임에 이루어지지 않음으로서 런타임에 오류가 발생하거나 또는 오류 사실을 알지 못하는 경우가 발생합니다.

하지만, 제네릭을 사용하면 컴파일러는 제네릭 클래스가 건네받은 타입 매개변수만을 사용하는 것을 보장해줍니다. 이것을 타입안정성이라고 합니다.

제네릭 클래스에서 설명한 FruitBox 클래스를 가져왔습니다.

```java
class FruitBox<T> {
    private T fruit;
    
    public FruitBox(T fruit) {
        this.fruit = fruit;
    }
    
    public T getFruit() {
        return fruit;
    }
}
```

제네릭 클래스의 인스턴스에서 아까와 같은 상황이 일어났다고 가정해봅시다.

```java
public static void main(String[] args) {
    FruitBox appleBox = new FruitBox(new Apple());
    FruitBox bananaBox = new FruitBox(new Banana());

    Apple apple = appleBox.getFruit();
    Banana banana = appleBox.getFruit(); // 컴파일 에러
}
```

컴파일 에러가 발생하는 것을 볼 수 있습니다. 자바 컴파일러는 제네릭 코드에 강한 타입 체크를 합니다. 그리고 만약 코드가 타입 안정성에 위배된다면 에러를 발생시킵니다. 컴파일 타임 에러를 수정하는 것은 발견하기 어려울 수 있는 런타임 에러를 수정하는 것보다 훨씬 쉽습니다.

또 하나의 장점은 캐스팅이 사라졌다는 겁니다. Object 클래스는 사용하는 타입에 맞게 캐스팅을 해줘야만 합니다. 하지만 제네릭은 컴파일 타임에 미리 타입이 정해지므로, 이런 형변환을 하지 않아도 됩니다.

요약하자면 제네릭은 다음과 같은 장점을 가집니다.

1. 컴파일 타임에 논리적인 자료형의 오류에 대한 검증을 수행하여 런타임에 자료형에 안전한 코드를 실행합니다.
2. 반환값에 대한 타입 변환 및 타입 검사에 들어가는 노력을 줄일 수 있고, 형변환이 없어지므로 가독성이 좋아집니다.

### 3. 제네릭 메소드

제네릭 메소드는 타입 매개변수를 반환 타입 전에 지정해주면 됩니다.

```java
public <T> void add(T t) {
	...
}
```

여기서 <T> 는  앞으로 add()메소드에게 T라는 타입 매개변수를 사용하겠다고 알려주는 것입니다. 이것으로 add() 메소드는 내부에서 타입 매개변수 T를 사용할 수 있습니다. 이 타입 매개변수의 범위는 메서드의 블록 이내로 한정됩니다.

제네릭 메소드는 제네릭 클래스뿐만 아니라, 제네릭 클래스가 아니어도 정의가 가능합니다. 즉, 제네릭 메소드는 제네릭 클래스에 독립적으로 정의할 수 있습니다. 여기서 주의할 점이 있습니다. 그것은 바로 제네릭 메소드의 타입 매개변수와 제네릭 클래스의 타입 매개변수는 다를 수 있다는 것입니다.

코드로 예제를 보겠습니다. ClassPrinter 클래스는 printClassName 이라는 하나의 메소드만을 갖고 있습니다. 이 메소드는 전달받은 매개변수의 클래스 이름을 출력합니다. ClassPrinter의 실제 타입 매개변수를 String으로 생성한 후, printClassName 메소드에 Integer 클래스와 Double 클래스의 인스턴스를 전달해보겠습니다.

```java
public class ClassPrinter<T> {
	public <T> void printClassName(T t) {
		System.out.println(t.getClass().getName());
	}

	public static void main(String[] args) {
		ClassPrinter<String> printer = new ClassPrinter<>();
		printer.printClassName(1);
		printer.printClassName(3.14);
	}
}

// 결과
java.lang.Integer
java.lang.Double
```

실행 결과, 클래스의 실제 타입 매개변수 String이 아닌 메소드 인자의 타입인 Integer와 Double이 나왔습니다. 즉, 메소드와 클래스의 타입 매개변수가 같다면, 타입 매개변수는 메소드 것을 따라갑니다.

### 4. 타입 매개변수의 제한

알다시피 A 객체가 B 객체의 상위 클래스라면, B 객체를 A 객체라고 할 수 있습니다. 예를 들어, Integer 객체를 Object 객체라고 할 수 있습니다.

```java
Object someObject = new Object();
Integer someInteger = new Integer(10);
someObject = someInteger;   // OK
```

그러므로 다음 메소드에서 Number는 Integer, Double의 상위 클래스이므로 사용가능합니다.

```java
public void someMethod(Number n) { /* ... */ }

someMethod(10);   // OK
someMethod(10.1);   // OK
```

이것은 제네릭에서도 똑같습니다. Number를 타입 매개변수로 갖고 있는 제네릭 타입은 만약 Number가 인자의

상위 클래스라면 호출 가능합니다.



```java
public static void main(String[] args) {
    List<Number> numbers = new ArrayList<>();
    numbers.add(new Integer(10));   // OK
    numbers.add(new Double(1.0));  // OK
}

class ArrayList<E> ... {

	...

	public boolean add(E e) {
		...
	}
}
```

하지만 다음과 같은 코드는 불가능합니다.

```java
public void addAll(List<Number> numbers) { /* ... */ }

List<Number> numbers = new List<>();
List<Integer> ints = new List<>();
numbers.addAll(ints); // ERROR
```

왜냐하면, List<Integer>는 List<Number>의 하위 클래스가 아니기 때문입니다.

![https://s3-us-west-2.amazonaws.com/secure.notion-static.com/e69bd116-05b0-4bc5-b417-e3d83454d889/Untitled_Diagram_(3).png](https://s3-us-west-2.amazonaws.com/secure.notion-static.com/e69bd116-05b0-4bc5-b417-e3d83454d889/Untitled_Diagram_(3).png)

Number 클래스와 Integer 클래스는 서로 상속관계이지만, List<Number>와 List<Integer>는 아무런 연관이 없습니다. List<Number>의 하위 클래스는 List<Integer>가 아닌 LinkedList<Integer> 혹은 ArrayList<Integer> 입니다. 즉, 상속관계는 타입 매개변수를 따라가는 것이 아닌, 앞의 클래스를 따라갑니다.

ArrayList<E> 는 List<E>를 구현합니다. List<E>는 Collection<E>를 상속받습니다. 그러므로 ArrayList<Integer>은 List<Integer>, Collection<Integer>의 서브타입입니다.

![https://s3-us-west-2.amazonaws.com/secure.notion-static.com/dd7ffaff-ee95-48a9-8d67-71defa2e7748/Untitled_Diagram_(4).png](https://s3-us-west-2.amazonaws.com/secure.notion-static.com/dd7ffaff-ee95-48a9-8d67-71defa2e7748/Untitled_Diagram_(4).png)

다음과 같은 음식에 대한 클래스 계층 구조가 있습니다.

Fruit 클래스와 Vegetable 클래스는 Food 클래스를 상속받습니다.

![https://s3-us-west-2.amazonaws.com/secure.notion-static.com/4e5ca87c-58ba-4c55-b4ea-bceef12d213f/Untitled_Diagram.png](https://s3-us-west-2.amazonaws.com/secure.notion-static.com/4e5ca87c-58ba-4c55-b4ea-bceef12d213f/Untitled_Diagram.png)

그리고 FruitBox 클래스가 있습니다. FruitBox 클래스는 Fruit 클래스의 리스트를 가지고 있습니다. add() 메소드로 해당 타입을 리스트에 추가할 수 있습니다.

```java
class FruitBox<T> {
    private List<Fruit> fruits = new ArrayList<>();
    
    public void add(T fruit) {
        fruits.add(fruit);
    }
}
```

Apple, Banana, Fruit은 모두 Fruit의 하위 클래스이므로, 위에서 보았던 것처럼 fruits에 원소들을 더할 수 있을 것입니다. 하지만, 정상적으로 보이는 이 코드는 컴파일 에러가 발생합니다. 왜냐하면 T가 Fruit의 하위 클래스가 아닐 수도 있기 때문입니다.

위 상속 계층도에서의 Vegetable 클래스가 만약 들어갔다고 해봅시다. 그럼 FruitBox는 다음과 같이 변한다고 생각할 수 있습니다.

```java
class FruitBox<Vegetable> {
    private List<Fruit> fruits = new ArrayList<>();
    
    public void add(Vegetable fruit) {
        fruits.add(fruit);
    }
}
```

과일 상자에 야채가 들어가버렸습니다. 이렇듯 잘못된 타입이 들어올 수 있으니 컴파일러가 경고를 보여주는 것입니다.

그러므로 Fruit의 하위 클래스만 들어올 수 있도록 경계, 바운드를 설정해주어야 합니다. 이것을 타입 매개변수의 제한이라고 합니다.

타입 매개변수의 제한에는 2가지 종류가 있습니다.

- T extends Fruit(Upper Bound) : T는 반드시 Fruit을 상속받은 객체 혹은 Fruit 클래스 그 자체여야 합니다.

그림으로 살펴보면 다음과 같습니다.

![https://s3-us-west-2.amazonaws.com/secure.notion-static.com/bd0fd989-90b3-421c-9b08-5666927e0242/Untitled_Diagram_(1).png](https://s3-us-west-2.amazonaws.com/secure.notion-static.com/bd0fd989-90b3-421c-9b08-5666927e0242/Untitled_Diagram_(1).png)

- T super U(Lower Bound) : T는 반드시 Fruit의 상위 클래스 혹은 Fruit 클래스 그 자체여야 합니다.

![https://s3-us-west-2.amazonaws.com/secure.notion-static.com/38037b42-7853-4179-a1f2-6456c69f8a40/Untitled_Diagram_(2).png](https://s3-us-west-2.amazonaws.com/secure.notion-static.com/38037b42-7853-4179-a1f2-6456c69f8a40/Untitled_Diagram_(2).png)

주의할 점으로 타입 매개변수의 경계 설정은 리턴 타입 앞에서 해야합니다.

```java
public <T> void print(List<T extends Fruit> list) { } // x
public <T extends Fruit> void print(List<T> list) { } // o
```

그러므로 위의 FruitBox 클래스에 Upper Bound를 설정해주면 Fruit의 하위 클래스만이 들어옴을 보장해줍니다.

```java
class FruitBox<T extends Fruit> {
    private List<Fruit> fruits = new ArrayList<>();
    
    public void add(T fruit) {
        fruits.add(fruit);
    }
}
```

컴파일 에러가 발생하지 않는 것을 확인할 수 있습니다.

여러 타입 매개변수를 지정할 수도 있습니다. 만약 Fruit의 하위 클래스이면서 Comparable의 하위 클래스이어야 한다면, 다음과 같이 나타낼 수 있습니다.

```java
class FruitBox<T extends Fruit & Comparable<T>> { ... }
```

이렇게 경계의 타입 매개변수가 여러 개 일 경우에는 무조건 Class 가 가장 먼저 와야 합니다. 다음과 같이 Comparable이 먼저 온다면 컴파일 에러가 발생합니다.

```java
class FruitBox<T extends Comparable<T> & Fruit> { ... } // 컴파일 에러 발생
```

또한, 자바는 다중 상속이 불가능하므로 경계의 타입 매개변수의 클래스는 최대 한 개만 존재할 수 있습니다.

# 5. Wildcards

다음은 와일드 카드입니다.

와일드 카드는 클래스의 타입 매개변수를 제외한 모든 곳에서 사용할 수 있습니다.

![https://s3-us-west-2.amazonaws.com/secure.notion-static.com/3d6a4958-a0d6-4575-af0e-414df848e892/_2021-02-13__6.34.47.png](https://s3-us-west-2.amazonaws.com/secure.notion-static.com/3d6a4958-a0d6-4575-af0e-414df848e892/_2021-02-13__6.34.47.png)

와일드 카드에는 세 가지 타입이 있습니다.

- Unbounded Wildcards
- Upper Bounded Wildcards
- Lower Bounded Wildcards

차례대로 얘기하겠습니다.

## Unbounded Wildcards

unbounded wildcard 타입은 `?` 문자를 사용하여 선언합니다. 예를 들어, `List<?>`  가 있습니다. 이것을 리스트의 unknown type이라고 부릅니다.

Unbounded Wildcard가 유용한 2가지 경우가 있습니다.

- Object 클래스의 메소드를 사용하여 메소드를 구현하는 경우
- 타입 파라미터에 의존하지 않는 제네릭 클래스의 메소드를 사용할 때. 예를 들어, List.size 혹은 List.claer.

다음의 printList 메소드를 보겠습니다.

```java
public static void printList(List<Object> list) {
    for (Object elem : list)
        System.out.println(elem + " ");
    System.out.println();
}
```

`printList` 의 목표는 리스트의 어떤 타입이든 출력하는 것입니다. 하지만 이 코드는 실패합니다. 왜냐하면 Object 리스트의 인스턴스만을 인자로 받기 때문입니다. 위에서 설명했듯이 List<Object>와 List<String>은 상속관계가 아닌 전혀 다른 클래스입니다. 그러므로 List<String>를 인자로 전달한다면 해당 메소드는 컴파일 에러를 띄웁니다.

어떤 타입이든 프린트하기 위해선 다음과 같은 `List<?>` 를 사용합니다.

```java
public static void printList(List<?> list) {
    for (Object elem: list)
        System.out.print(elem + " ");
    System.out.println();
}
```

임의의 타입 A의 리스트 List<A>는 List<?>의 서브 타입입니다. 그러므로 어떤 타입의 리스트이든 printList 할 수 있습니다. `List<String>` 은 `List<Object>`의 서브 타입이 아닙니다. 하지만 `List<String>` 은 `List<?>` 의 서브 타입입니다.

### List<?> 에는 null만 삽입할 수 있다

왜냐하면 해당 List에는 어떠한 값이 들어있을지 모르기 때문입니다. 예를 들어, 다음과 같은 인트형 리스트가 있습니다. 그 뒤로, List<?> 타입을 받아서 그 리스트에 더블을 추가하는 메소드가 있습니다.

```java
public static void main(String[] args) {
    List<Integer> ints = new ArrayList<>();
    addDouble(ints);
}

private static void addDouble(List<?> ints) {
    ints.add(3.14);
}
```

만약, List<?>가 값을 넣는 것을 허용한다면 List<Integer>에 Double형이 들어가게 됩니다. 이것은 제네릭의 타입 안정성에 위배되는일입니다. 그러므로, List<?>에는 `null`만을 삽입할 수 있습니다.

### List<?>에서 get한 원소는 Object이다

비경계 와일드카드의 원소는 어떤 타입도 될 수 있기 때문에, 꺼낸 원소 타입은 가장 범용적인 형인 Object 입니다.

```java
public void bar(List<?> list) {
    Object o = list.get(0);
    String s = list.get(0); //コンパイルエラー
}
```

## Upper Bounded Wildcards

변수의 제한을 유연하게 하기 위해 upper bounded wildcard를 사용할 수 있습니다. Upper Bounded wildcard 는 `? extends ...` 의 형태로 사용합니다. 위에서 사용한 FruitBox 클래스를 예제로 살펴보겠습니다.

다음은 위에서 본 FruitBox 클래스입니다.

```java
class FruitBox<T extends Fruit> {
    private List<Fruit> fruits = new ArrayList<>();
    
    public void add(T fruit) {
        fruits.add(fruit);
    }
}
```

여기에 새로운 기능을 추가하려고 합니다. 바로 다른 과일 박스에 있는 과일들을 가져와서 현재 과일 박스에 추가하는 기능입니다.

```java
class FruitBox<T extends Fruit> {
    private List<Fruit> fruits = new ArrayList<>();
    
    public void add(T fruit) {
        fruits.add(fruit);
    }

		public void addAll(FruitBox<Fruit> fruitBox) {
        this.fruits.addAll(fruitBox.fruits);
    }
}
```

다음과 같이 addAll() 메소드를 추가했습니다. 이 메소드는 List<Fruit>을 인자로 받아서 현재 과일의 리스트에 추가합니다. 하지만 이 코드는 한 가지 문제점이 있습니다.

바로 다음 코드에서 컴파일 에러가 발생한다는 것입니다.

```java
public static void main(String[] args) {
    FruitBox<Fruit> fruitBox = new FruitBox<>();
    
    FruitBox<Apple> appleBox = new FruitBox<>();
    appleBox.add(new Apple());
    
    fruitBox.addAll(appleBox); // Compile Error
}
```

왜냐하면, 위에서 말했다시피 FruitBox<Fruit>과 FruitBox<Apple>은 어떤 연관 관계도 없기 때문입니다. 즉, 현재 메소드는 Fruit 클래스만을 받을 수 있다는 것입니다. Fruit 클래스를 포함한 하위 클래스까지 받을 수 있게 해주는 것이 Upper Bounded Wildcards 입니다.

위 메소드에 Upper Bounded Wildcards를 추가한 코드입니다.

```java
public void addAll(FruitBox<? extends Fruit> fruitBox) {
    this.fruits.addAll(fruitBox.fruits);
}
```

같은 메소드를 작성해보면 컴파일 에러가 발생하지 않는 것을 확인할 수 있습니다.

```java
public static void main(String[] args) {
    FruitBox<Fruit> fruitBox = new FruitBox<>();
    
    FruitBox<Apple> appleBox = new FruitBox<>();
    appleBox.add(new Apple());
    
    fruitBox.addAll(appleBox);
}
```

### Wildcard Guidelines

- `in` variables는 상한 와일드카드로 정의되어야 한다.
- `out` variables는 하한 와일드카드로 정의되어야 한다.
- Object 클래스에 정의된 메소드를 사용하여 `in` 변수에 접근할 수 있는 경우 Unbounded 와일드카드를 사용한다.
- `in` , `out` 변수 둘 다 접근해야 한다면 와일드 카드를 사용하면 안된다.

이 가이드라인은 메소드 리턴 타입에는 적용하지 않습니다. 리턴 타입에 와일드 카드를 사용하는 것은 피해야 합니다.  그것은 프로그래머가 코드를 사용하여 와일드카드를 처리하도록 합니다.

List<? extends ...?> 은 read-only로 생각할 수 있습니다. 엄격한 의미에서는 read-only가 아닙니다. 하지만 새 요소를 저장하거나 기존 요소를 변경할 수 없기 때문에 그렇게 생각할 수 있습니다.

```java
public void wildcardHasAnyType() {
    List<String> stringList 
			= new ArrayList<>();

    List<Integer> integerList 
			= new ArrayList<>();

    boolean b1 = contains(stringList, "a");
    boolean b2 = contains(integerList, 3);
}

boolean contains(List<?> list, Object o) {
    return list.contains(o);
}
```

List<?>는 어떤 클래스의 원소를 가진 리스트를 나타냅니다. List<?>에는 임의의 List를 대입할 수 있습니다.

이것은 공변의 성질에 의해 List<Object>에는 List<String>을 도입할 수 없었던 것과 대조적입니다. 이렇게 와일드카드를 이용하면 제네릭의 불변 제약을 느슨하게 할 수 있습니다.

### List<? extends T>는 공변성을 가진다

List<? extends T>는 상한적 와일드카드 타입입니다. 이것은 타입 T의 어떤 서브클래스를 원소로 하는 List를 나타냅니다. 상한적 와일드 클래스 타입을 사용하면 다음 코드와 같이, 불변이었던 제네릭에 공변성을 도입할 수 있습니다.

```java
public void foo() {
    List<Number> numList = new ArrayList<>();
    List<? extends Number> wildCardNumList = new ArrayList<>();
    List<Integer> intList = new ArrayList<>();

    wildCardNumList = intList; //OK
    numList = intList; //コンパイルエラー
}
```

### List<? super T>는 반변성을 가진다

List<? super T>는 하한적 와일드 카드입니다. 이것은 타입 T의 어떤 슈퍼 클래스를 원소로 하는 List라는 의미입니다. 하한적 와일드 카드를 사용하면 아래 코드처럼, 불변이었던 제네릭에 반변성을 도입할 수가 있습니다.

```java
public void bar() {
    List<Integer> intList = new ArrayList<>();
    List<? super Integer> wildCardIntList = new ArrayList<>();
    List<Number> numList = new ArrayList<>();

    wildCardIntList = numList; //OK
    intList = numList; //コンパイルエラー
}
```

지정된 제네릭 타입은 유연하지 않고 재사용하기 어렵습니다. 그래서 나온 것이 `wildcard` 입니다.

제한되지 않은 와일드카드는 메소드에서 formal 매개변수를 선언할 때 더 중요합니다.

![https://s3-us-west-2.amazonaws.com/secure.notion-static.com/57d2744c-cce6-48c8-82c5-2d995b770bf2/Untitled.png](https://s3-us-west-2.amazonaws.com/secure.notion-static.com/57d2744c-cce6-48c8-82c5-2d995b770bf2/Untitled.png)

위 코드에서 runOne 메소드는 오직 지정된 타입 `Animal` 클래스만을 매개변수로 받는 것을 알 수 있습니다.  그리고 runTwo 메소드는 모든 타입을 매개변수로 받을 수 있습니다. 그러므로 더 유연합니다.

extends 키워를 사용한 곳에서 Animal 클래스의 하위 클래스들을 파라미터로 넘겨줄 수 있습니다.

### super

파라미터화된 타입(제네릭 클래스)는 반드시 지정한 타입 혹은 그 상위클래스여야 합니다.

![https://s3-us-west-2.amazonaws.com/secure.notion-static.com/d24b4c98-dabd-451c-9c1b-3336efcab7da/Untitled.png](https://s3-us-west-2.amazonaws.com/secure.notion-static.com/d24b4c98-dabd-451c-9c1b-3336efcab7da/Untitled.png)

List<? extends Fruit> 클래스를 사용하여, Fruit 클래스 혹은 그 하위 클래스를 지정한 List형으로 선언 가능합니다.

```java
List<? extends Fruit> basket = new ArrayList<Fruit>(); // 1
List<? extends Fruit> basket = new ArrayList<Melon>(); // 2
List<? extends Fruit> basket = new ArrayList<Lemon>(); // 3
```

다음으로 baseket에서 원소를 꺼낼 떼, 그 서브 클래스를 지정할 수 있는가를 생각해보겟습니다.

List<? extends Fruit> 은 적어도 아래 3개의 List로 변환 가능합니다.

```java
1. ArrayList<Fruit>
2. ArrayList<Melon>
3. ArrayList<Lemon>
```

따라서 원소를 꺼낼 때, 몇 가지 계열에 파생되어 있는 서브 클래스를 지정할 수가 없습니다.

```java
List<? extends Fruit> basket = ...

// NG 컴파일 에러, 멜론형으로 변환 불가
Melon fruit = basket.get(i);
```

만약 `? extends Fruit` 을 서브 클래스로 변환하려고 했다면, 아래와 같은 모순이 발생한다.

```java
List<Melon> melonList = new ArrayList<Melon>();
melonList.add(new Melon());

List<? extends Fruit> basket = melonList;

// 서브 클래스로의 변환이 가능하다면 Lemon으로 가져올 수 있다
Lemon fruit = basket.get(i);
```

따라서 서브 클래스로의 변환이 허용되지 않습니다.

따라서 `? extends T` 에서 T의 서브 클래스로의 변환은 불가능합니다.

다음으로 `List<? extends Fruit> fruits` 로 원소를 추가하는 것을 생각해봅시다.

`? extends Fruit` 에서 `Fruit`으로 변환은 가능하지만, 다음과 같이 `fruits` 에 `Fruit` 오브젝트를 추가할 수 있는 것처럼 보입니다.

하지만 다음을 허용한다면 아래와 같이 모순이 발생합니다.

```java
List<? extends Fruit> fruits = new ArrayList<Melon>();
Fruit fruit = new Lemon();
fruits.add(fruit);
```

따라서, 구체적인 T 혹은 T의 서브 클래스로부터 `? extends T` 로의 변환은 불가능합ㄴ디ㅏ.

이것은 List 형을 상한적 와일드카드로 선언하면, 원소를 추가하는 것이 불가능하다는 것을 의미합니다.

### 하한적 와일드 카드

```java
Generic<? super T>
```

타입 매개변수를 위와 같이 지정하면 그 파라미터는 하한적 와일드 카드형입니다.

`? super T` 형은 T 혹은 T의 상위 클래스 전부를 의미합니다.

![https://s3-us-west-2.amazonaws.com/secure.notion-static.com/6a07505d-a91e-43c2-bd70-29f5b00ac795/Untitled.png](https://s3-us-west-2.amazonaws.com/secure.notion-static.com/6a07505d-a91e-43c2-bd70-29f5b00ac795/Untitled.png)

`List<? super Melon>` 은 Fruit 클래스 혹은 그 상위 클래스로 타입 매개변수를 선언할 수 있습니다.

```java
List<? super Melon> baskets = new ArrayList<Melon>(); // 1
List<? super Melon> baskets = new ArrayList<Fruit>(); // 2
List<? super Melon> baskets = new ArrayList<Food>(); // 3
List<? super Melon> baskets = new ArrayList<Object>(); // 4
```

### 6. Type Erasure

제네릭을 구현하면, 자바 컴파일러는 타입 이레이져를 적용합니다.

타입 이레이져는 다음과 같은 일을 합니다.

- 제네릭 타입의 모든 타입 파라미터를 경계 타입, 만약 경계가 없다면 Object로 치환합니다. 그러므로, 생성된 바이트코드는 오직 기존의 클래스, 인터페이스, 메소드만을 가집니다.
- 만약 타입 안정성을 유지해야 한다면 캐스팅을 삽입합니다.

타입 이레이져는 파라미터화된 타입에 대한 새로운 클래스를 생성하지 않는 것을 보장해주빈다. 결과적으로 제네릭은 런타임 오버 헤드를 발생시키지 않습니다.

타입 이레이져는 컴파일 타임에만 타입 제약을 적용하고, 런타임에는 타입 정보를 버리는 절차를 의미합니다.

매개변수화 타입, 즉 타입 매개변수가 가진 클래스 정보는 컴파일러에 의해 없어집니다. 이것을 `Type Erasure` 라고 부릅니다.

다음과 같은 제네릭 Box 클래스가 있습니다.

```java
public class Box<T> {
    private List<T> objects = new ArrayList<>();
    
    public void add(T object) {
        objects.add(object);
    }
    
    public T get(int index) {
        return objects.get(index);
    }
}
```

컴파일러는 다음과 같이 정규 타입 매개변수를 Object로 변경합니다.

```java
public class Box {

   private List objects = new ArrayList();

   public void add(Object object) {
      this.objects.add(object);
   }

   public Object get(int index) {
      return this.objects.get(index);
   }
}
```

그러므로 컴파일러는 타입 안정성을 보장하고, 런타임 에러를 방지합니다.

Object 타입으로 가지고 있다가 사용할 때 Casting 하여 원하는 클래스로 반환한다.

```java
public class generic/Box {

	private Ljava/util/List; objects

	//	public 메소드명(파라미터 타입) 반환 타입

  public add(Ljava/lang/Object;)V // 파라미터의 타입이 T가 아닌 Object
		...
    INVOKEINTERFACE java/util/List.add (Ljava/lang/Object;)Z (itf)
		...

  public get(I)Ljava/lang/Object; // 반환 타입이 T가 아닌 Object
		...
    INVOKEINTERFACE java/util/List.get (I)Ljava/lang/Object; (itf)
		...
}
```

런타임에는 형식 타입(Formal Type)은 Bound 되었을 경우는 바운드 타입으로, Bound 되지 않았을 경우에는 Object로 바뀝니다.

예를 들어, 아래와 같은 형변환  T를 가진 제네릭 클래스가 있습니다.

Container는 Integer나 BigDecimal 등의 Number 클래스를 갖는 클래스입니다.

```java
class Container<T extends Number> {
    private T value;    
    public Container(T value) {
        this.value = value;
    }   
    public T get() {
        return value;
    }
}
```

컴파일러는 위의 클래스에서 타입 정보를 삭제하여, 아래의 클래스와 같이 만듭니다.

```java
class Container {
    private Number value;
    public Container(Number value) {
        this.value = value;
    }
    public Number get() {
        return value;
    }
}
```

타입 정보를 실제로 삭제하는 것은 아니고, 타입 매개변수는 그 상한경계의 타입(위에선 Number)로 변경됩니다. 이렇게 타입 정보는 컴파일러에 의해 소거되기 때문에, 실행시에는 타입 정보를 취득할 수 없습니다.

타입 이레이져는 제네릭을 사용하는 Java 애플리케이션이 제네릭이 도입되기 전에 생성된 Java 라이브러리 및 애플리케이션과 바이너리 호환성을 유지하도록 해줍니다.

예를 들어, Box<String>은 Box라는 raw type으로 변형됩니다. raw type은 타입 매개변수를 제외한  제네릭 클래스 혹은 인터페이스의 이름입니다. 이건 런타임에 사용되고 있는 제네릭 클래스의 실제 타입 매개변수를 찾을 수 없다는 것을 의미합니다.

그러므로 다음 연산은 불가능합니다.

```java
public class MyClass<E> {
    public static void myMethod(Object item) {
        if (item instanceof E) {  //Compiler error
            ...
        }
        E item2 = new E();       //Compiler error
        E[] iArray = new E[10];  //Compiler error
        E obj = (E)new Object(); //Unchecked cast warning
    }
}
```

위 코드는 런타임에 의미가 없습니다. 왜냐하면 컴파일러가 컴파일 타임에 실제 타입 매개변수에 대한 모든 정보를 지우기 때문입니다.

제네릭을 사용하지 않던 코드와 제네릭 코드와의 호환성을 위해 타입 이레이져가 존재합니다. raw type을 사용는 것은 어떤 이유에서든 나쁜 프로그래밍 방법으로 고려되니, 가능하면 피해야만 합니다.

이에 따라서 하나 더 알아두시면 좋을게 있는데요.

바로 타입 매개변수의 클래스를 생성할 수 없다는 것입니다.

즉, `new T()` 할 수 없습니다.

타입 파라미터는 소거되기 때문에, T의 인스턴스를 생성할 수 없습니다.

마찬가지로 `new T[]` 도 할 수 없습니다.

제네릭 클래스의 경우, 다음과 같이 매개변수 T를 타입으로 하는 인스턴스를 생성할 수 있습니다.

```java
class Sample<T> {
    public List<T> createList(int size) {
        //TのArrayListを生成せいせいできる
        return new ArrayList<T>(size);
    }
}
```

하지만, 타입 매개변수를 요소로 하는 배열을 생성할 수 없습니다.

```java
class Sample<T> {   
    public T[] createArray(int size) {
        return new T[size];
    }   
}
```

위에서 설명한대로, 타입 매개변수 T의 클래스 정보는 실행시에 소거되기 때문입니다. 제네릭 클래스의 경우, 클래스 정보를 포함하지 않는 원형을 생성하기 때문에, 실행시에 클래스 정보를 필요로 하지 않습니다. 한편, 배열의 생성에는 실행시에 클래스 정보가 필요하기 때문에(종학히는 배열을 생성하는 바이트코드가 클래스를 필요로 합니다), 타입 맥변수 T를 원소로 하는 배열은 생성할 수 없습니다.

```java
public class Generic {
    public static void main(String[] args) {
        int[] ints = new int[5];
        String[] strings = new String[10];
    }
}
```

바이트 코드


제네릭을 정의했다면, 해당 타입이 기본 타입입니다.

오리지널 타입의 클래스 이름은 타입 매개변수를 제외한 제네릭 클래스의 이름입니다.

즉, `List<String>` 을 정의했다면, 오리지널 타입 클래스의 이름은 `List` 입니다.

타입 제한이 있을 경우 타입 매개변수를 삭제합니다. 첫 번째 제한된 타입들로 교체됩니다(그러므로 여러 타입들이 T로 제한될 수 있습니다). 제한되지 않은 변수는 텍스트를 읽을 때 명확하지 않을 수 있는  `Object` 클래스를 사용합니다.

제네릭 코드 안에서 타입 매개변수에 대한 어떠한 정보도 얻을 수 없습니다. 객체를 생성할 때의 타입 파라미터의 타입(actual type)을 알 수 없습니다.

요약해서 자바 제네릭은 Erase를 사용합니다. 이것은 제네릭을 사용할 대, 어떤 지정 타입 정보도 지워진다는 것을 의미합니다. 알아야 할 것은 Object를 사용하고 있다는 것입니다. 그러므로 `List` 와 `List`는 실제로는 런타임시 동일한 타입이라는 것입니다.

1.5부터 제네릭이 도입되었다.

1.5 이전에는 제네릭을 사용하지 않았다는 것이다.

그러면 1.5 이전 버전에서 작성한 코드를 1.5버전에서는 사용할 수 없는 것일까?

코드로 살펴보자.

1.5 이전 버전에서 리스트는 다음과 같이 생성하였다.

```java
List fruits = new ArrayList();
```

1.5 부터 제네릭을 사용하는 코드에서는 다음과 같이 생성한다.

```java
List<Fruit> fruits = new ArrayList<>();
```

1.5 버전부터 List는 제네릭을 사용한다. 제네릭을 사용하지 않은 1.4 버전의 코드는 1.5버전에서 작동할끼? 1.5 이전버전과 1.5버전부터의 호환성을 위해 나타난 것이 바로 Type erasure이다.

### 제네릭의 제거 시기

자바 코드에서 선언되고 사용된 제네릭 타입은 컴파일 시 컴파일러에 의해 자동으로 검사되어 타입 변환됩니다.

그리고서 코드 내의 모든 제네릭 타입은 제거되어, 컴파일된 class 파일에는 어떠한 제네릭 타입도 포함되지 않게 됩니다.

이런 식으로 동작하는 이유는 제네릭을 사용하지 않는 코드와의 호환성을 유지하기 위해서입니다.

### 제네릭 클래스는 staic문(static 변수, static 메소드, static 초기화 블럭)에 사용할 수 없다.

정규 타입 매개변수의 실제 타입 매개변수는 외부에서 제네릭 클래스 타입의 레퍼런스를 선언하거나 인스턴스를 생성할 때 결정이 되는데, static문들든 인스턴스의 생성과는 무관하게 존재하므로 실제 타입을 알 수 없다.

### 제네릭에서 Primitive Type을 사용할 수 없는 이유

제네릭에서 다음과 같이 타앱 매개 변수로 Primitive Type을 사용할 수 없다.

```java
List<int> list = new ArrayList<>(); // 컴파일 에러
```

어째서 primitive type이 작동하지 않는지 알기 위해선, 먼저 제네릭은 컴파일 타임 기능이라는 것을 기억해야 한다. 타입 매개변수는 지워지고, 모든 제네릭 타입은 Object 타입이 됩니다.

```java
// 제네릭 클래스의 메소드
boolean add(E e)

// 컴파일 후
boolean add(Object e)
```

그러므로 타입 매개변수는 반드시 Object로 바뀔 수 있어야 합니다. 하지만 primitive type은 Object를 상속받지 않았기 때문에 primitive 타입을 타입 매개변수로 사용할 수 없습니다.

그러나, 자바는 오토박싱과 언박싱과 함께 primitive type에 대한 박싱된 타입을 제공합니다.  그러므로 만약 Integer 클래스를 담은 리스틀 만들고 싶다면, 이 wrapper 클래스를 이용합니다.

```java
List<Integer> list = new ArrayList<>();
list.add(17);
int first = list.get(0);
```

컴파일된 코드는 다음과 같습니다.

```java
List list = new ArrayList<>();
list.add(Integer.valueOf(17));
int first = ((Integer) list.get(0)).intValue();
```

## 7. Restrictions on Generics

### 타입이 타입 파라미터인 스태틱 필드를 선언할 수 없다

클래스의 스태틱 필드는 클래스의 모든 non-static 객체들이 공유하는 클래스 레벨 변수입니다. 따라서, 타입 파라미터의 스태틱 필드는 허용되지 않습니다. 예를 들어, 다음과 같은 클래스가 있습니다.

```java
public class MobileDevice<T> {
    private static T os;

    // ...
}
```

만약 타입 파라미터의 스태틱 필드가 허용된다면, 아래의 코드는 혼란스럽게 됩니다.

```
MobileDevice<Smartphone> phone = new MobileDevice<>();
MobileDevice<Pager> pager = new MobileDevice<>();
MobileDevice<TabletPC> pc = new MobileDevice<>();
```

왜냐하면 스태틱 필드 `os` 는 `phone`, `pager`, `pc` 에서 공유되기 때문입니다. 실제 os의 타입은 무엇일까요? `os` 는 동시에 `Smartphone`, `Pager`, `TabletPC` 가 될 수 없습니다. 그러므로 타입 파라미터의 스태틱 필드를 선언할 수 없습니다.
